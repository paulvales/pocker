# 2026-03-21 Session

## Decisions

### Default room runtime moved back to memory
**What:** The app no longer uses PostgreSQL for room runtime state or audit by default. `createServerApp` now defaults to an in-memory room runtime store and an in-memory audit store, while PostgreSQL remains in use for estimation history only.
**Why:** The user reported 1-2 second delays on vote/reaction after the runtime state was moved to PostgreSQL and clarified that the database is only needed for persistent estimate history. Keeping room/session/vote runtime in memory restores the original low-latency socket behavior while preserving DB-backed history.
**Impact:** Local and default app runs no longer hit the remote `DATABASE_URL` for vote/reaction/session runtime updates. Explicit `roomRuntimeStoreOptions` or `auditLogStoreOptions` still opt into the PostgreSQL-backed stores for tests or special deployments. A new server regression test verifies that the default app can create/join/vote with `DATABASE_URL` removed while history remains Postgres-backed.

<details>
Context:
The runtime room state had been migrated to PostgreSQL, which caused every vote and membership touch to hit the remote Aiven database. The user observed noticeable local latency and clarified that Postgres should only be used for estimation history persistence.

Options considered:
- Keep the PostgreSQL runtime store and try to reduce per-vote DB traffic.
- Restore in-memory runtime/audit defaults and keep PostgreSQL only for history.

Decision:
Add `createInMemoryRoomRuntimeStore` and `createInMemoryAuditLogStore`, and change `createServerApp` to use them by default unless explicit `roomRuntimeStoreOptions` or `auditLogStoreOptions` are provided. Leave the existing PostgreSQL-backed store implementations available for opt-in tests and special cases.

Tradeoffs:
Default runtime state is again process-local, so it no longer provides cross-instance synchronization out of the box. That is acceptable for the requested architecture and restores responsiveness. Shared-runtime tests still use explicit pg-mem-backed options.

Follow-up:
If multi-instance runtime ever becomes necessary again, make it an explicit deployment mode instead of the default path.
</details>

## Bugs Fixed

### RoomPage gates admin UI and visible average
**What:** RoomPage now keeps `#averageVote1` live for automation while hiding the visible average until reveal, removes vote buttons from the pre-join room screen by not rendering them at all, hides the points/hours toggle from participants, and keeps the viewer current-task label as a real external link only when the task is a URL.
**Why:** The visible room UI had regressed from role/stage-based behavior to sharing one average value and leaving sensitive controls mounted in the DOM. Separating display concerns and guarding render paths directly is the more robust fix.
**Impact:** `RoomPage.tsx` now separates hidden and visible average values, uses conditional rendering for joined/admin-only controls, and renders the viewer task as an anchor only when a URL exists. `RoomPage.test.tsx` now covers hidden-average behavior, missing vote controls on join, viewer role gating, and the current-task external link.

<details>
Context:
The user reported four issues on the route-based room page: average score was visible before reveal, vote buttons were visible on the login screen, participants could see the admin points/hours toggle, and the participant current-task label should open URL tasks in a new tab.

Options considered:
- Keep the existing DOM mounted and rely on `display:none` plus a separate visible-average value.
- Switch the affected UI sections to conditional rendering and split hidden vs visible average values.

Decision:
Use conditional rendering for joined/admin/viewer-specific sections in `RoomPage.tsx`, keep `#averageVote1` bound to the live average for integrations, and render the visible average as `?` until reveal. Render `viewerTaskLink` as an anchor only when `getTaskHref` returns a real URL.

Tradeoffs:
Conditional rendering changes the DOM shape instead of only toggling styles, but it removes the chance of CSS or stale state leaking controls to the wrong role. The visible average now shows `?` before reveal instead of a numeric placeholder.

Follow-up:
If the user wants a different hidden placeholder text for the visible average, adjust `visibleAverageVoteValue` in `RoomPage.tsx` and extend the test accordingly.
</details>


### Admin seat now frees when room is empty
**What:** Joining as admin no longer fails with `ADMIN_ALREADY_EXISTS` after everyone has left the room. The room registry now treats a fully empty room as having a free admin seat, while still reserving the seat during recovery if any active participants remain.
**Why:** After restoring the default runtime store to memory, disconnected admin sessions still blocked the admin seat for the recovery TTL even when the room had no active participants. That made empty rooms incorrectly reject a new admin join.
**Impact:** `room-registry.js` now checks for active participants before applying admin-seat reservation logic in both join-time validation and `requestAdminStatus`. The server regression test now covers the empty-room case and verifies that a replacement admin can join immediately after the previous admin leaves.

<details>
Context:
The user reported that after all participants left a room, trying to join as admin still failed with `This room already has an active admin.`

Options considered:
- Shorten or remove the recovery reservation globally.
- Keep the recovery reservation only while there are active participants in the room.

Decision:
Update `room-registry.js` so admin-seat reservation is enforced only when `listActiveSessions` shows at least one active participant. If the room is empty, `requestAdminStatus` returns available and admin join is allowed immediately.

Tradeoffs:
A disconnected admin in an otherwise empty room no longer exclusively holds the seat during recovery TTL. This matches the requested behavior and removes the empty-room lockout.

Follow-up:
If the user wants different semantics for explicit leave vs transient disconnect, that would need a distinct session state beyond the current disconnect flow.
</details>
