### APP-12 moved to Aiven PostgreSQL
**What:** Estimate history persistence no longer uses a local JSON file. The app now stores history in PostgreSQL via DATABASE_URL, with the current local setup pointed at an Aiven instance and tests isolated through pg-mem.
**Why:** The user rejected free managed options that auto-pause on inactivity and also did not want another database container on a 2 GB host with many services. They chose Aiven PostgreSQL and provided a connection string, so the history store was migrated to a real relational backend while keeping local tests off the production database.
**Impact:** The server now initializes a PostgreSQL-backed estimation history table on startup, /api/estimation-history reads asynchronously from Postgres, reveal events persist to Postgres with upsert semantics by room/task/participant, docker-compose exposes DATABASE_URL to the app, and the local ignored .env file stores the current Aiven connection string. Jest server tests now inject pg-mem through a test-only history store override instead of using a JSON temp file.

<details>
Context: Estimate history originally used a file-backed JSON store. The user wanted a relational backend reachable externally and later selected Aiven PostgreSQL after ruling out auto-pausing free managed options and heavier self-hosted choices on a 2 GB server.

Options considered: Keep JSON or SQLite locally, use Supabase free, use Aiven free, self-host Postgres, or move to managed PostgreSQL. Managed PostgreSQL on Aiven was chosen because it avoids local RAM pressure and provides standard external SQL connectivity.

Decision: Replace the JSON store with a PostgreSQL store in estimation-history-store.js, driven by DATABASE_URL. Keep the runtime configuration out of git via a local .env file, pass DATABASE_URL through docker-compose, and use pg-mem in tests through a test-only global store options override.

Tradeoffs: Runtime history operations are now async and depend on database availability. The store strips sslmode from the connection string and applies ssl.rejectUnauthorized=false so the provided Aiven URL works with node-postgres without a local CA bundle. This is practical for the current setup but less strict than full certificate verification.

Follow-up: If the project later needs more persistent domain state beyond estimate history, reuse the same PostgreSQL instance and move room/session data into relational tables rather than adding another persistence layer.
</details>

## Bugs Fixed

### History overwrite now ignores room in unique key
**What:** Estimate history overwrite was broken for re-estimation in a different room because the PostgreSQL unique key still included room_id. The store now upserts by task_id + participant_name and updates room_id to the latest room.
**Why:** The user reported that overwrite behavior was broken after the PostgreSQL migration. The previous key `(room_id, task_id, participant_name)` only overwrote within the same room, which did not match the intended behavior for re-estimating the same task later from another room.
**Impact:** Repeated estimates for the same task and participant now replace the previous history row even across rooms, the latest room is preserved in the row, and the Aiven schema has been migrated from the old room-scoped unique index to `estimation_history_task_participant_unique` on `(task_id, participant_name)`.

<details>
Context: After moving APP-12 history storage to Aiven PostgreSQL, the app preserved room_id in the unique key used for upsert. That meant re-estimating the same task from a new room created a separate history row instead of replacing the old one.

Options considered: Keep room in the unique key and ask the user to reuse the same room for re-estimation, or change uniqueness to reflect the business rule that the latest estimate for a task/participant pair should win regardless of room. The latter matches the reported expectation and earlier user wording about overwriting corrected estimates.

Decision: Change the unique constraint and ON CONFLICT target from `(room_id, task_id, participant_name)` to `(task_id, participant_name)`, while still storing room_id as the latest room on the row. Add a migration in store initialization that drops the old constraint and adds the new one, and keep legacy deduplication enabled in real Postgres but disabled in pg-mem tests.

Tradeoffs: History no longer preserves multiple room-specific rows for the same task/participant pair; instead it preserves the latest state and room. This matches overwrite semantics but is less suitable if room-scoped historical versions are needed later.

Follow-up: If true multi-session history is ever needed, introduce a separate immutable history table plus a latest-state table instead of reusing one table for both concerns.
</details>


### History unique key now includes estimate type
**What:** Estimate history rows no longer let hours overwrite points for the same task and participant. The PostgreSQL unique key and upsert target now include estimate_type in addition to task_id and participant_name.
**Why:** After changing overwrite semantics to ignore room, the unique key became too broad and caused `hours` and `points` entries to collide. The user reported that hours were overwriting points, which meant estimate type had to become part of the identity of a history row.
**Impact:** The Aiven schema now uses `estimation_history_task_participant_type_unique` on `(task_id, participant_name, estimate_type)`, repeated estimates still overwrite within the same type, and points/hours for the same task coexist as separate history rows. Tests now cover both cross-room overwrite and cross-type separation.

<details>
Context: The prior fix changed overwrite semantics from room-scoped to task/participant-scoped so that re-estimating the same task in a different room replaced the old row. That accidentally made estimate type collisions possible.

Options considered: Keep the broader key and store only the latest estimate regardless of scale, or include estimate_type in the unique identity of a history row. Because the app explicitly supports both points and hours as separate estimation modes and the user reported incorrect overwrites, estimate_type had to be included.

Decision: Update the PostgreSQL table constraint and ON CONFLICT target from `(task_id, participant_name)` to `(task_id, participant_name, estimate_type)`. Drop the previous room-scoped and type-less constraints during initialization, and deduplicate legacy rows only by the new key in real Postgres. Add a regression test that records the same task once in points and once in hours and expects two rows.

Tradeoffs: Rows now represent the latest estimate per task/participant/type rather than a single latest estimate per task/participant. Previously overwritten cross-type data cannot be automatically reconstructed from the database.

Follow-up: If users later need a full audit trail instead of only latest-per-type state, add an immutable event table and keep the current table as a materialized latest-state view.
</details>
