# 2026-03-19 Session

## Decisions

### Added per-player emoji reactions
**What:** APP-11 adds a per-player reaction field synced through room state, validated against a fixed 7-emoji whitelist, and rendered as a large overlay on participant cards with a client dropdown.
**Why:** The app already syncs player state through the in-memory room registry and Socket.IO broadcasts, so reactions fit cleanly as another player attribute without adding a separate store or admin workflow.
**Impact:** Participants can choose one of 7 reactions from a dropdown, everyone in the room sees the reaction on the participant card, late joiners receive current reactions, and invalid emojis are rejected server-side.

<details>
Context: APP-11 required Discord-like participant reactions visible directly on each participant card.
Options considered: keep reactions purely client-side, store them separately from players, or add them to the existing player state object.
Decision: store a single `reaction` field per player in `room-registry`, validate via a fixed server whitelist, expose a `set_reaction` socket event with ack, and render the chosen emoji as a large overlay on the existing player card UI. Client re-applies a saved reaction after reload using the same TTL storage approach as votes.
Tradeoffs: the emoji list is duplicated in server and client for simplicity because the frontend is a static HTML file without a build step. Reactions persist across reveal/reset because they are treated as participant state, not round state.
Follow-up: if reactions need richer semantics later, extract the emoji list to a shared config surface and decide whether reset should also clear reactions.
</details>


### Reaction picker moved to floating messenger UI
**What:** APP-11 reaction UX now uses a fixed bottom-left floating button with an emoji-only popover, and card reactions overlay the vote area while keeping the player name visible.
**Why:** The first dropdown-based implementation met sync requirements but not the desired UX. A Telegram-like floating picker is closer to the requested interaction model and keeps reaction controls out of the main layout.
**Impact:** Users pick reactions from a floating bottom-left emoji tray, clicking the active emoji clears it, reactions animate on the card when changed, and the emoji visually replaces the vote area instead of covering the name.

<details>
Context: The initial APP-11 implementation added reactions with a standard dropdown under the controls. The follow-up requirement asked for a messenger-style picker, bottom-left fixed trigger, emoji-only options, and card overlay behavior that preserves the participant name.
Options considered: keep the inline selector and restyle it, place the trigger inside each card, or add a fixed floating picker for the current user. 
Decision: replace the inline selector with a fixed floating trigger in the bottom-left corner and a pill-shaped emoji tray that opens like a Telegram-style picker. Card markup was restructured into a body/footer split so the reaction overlays the vote zone while the player name remains visible below. A CSS animation is triggered only when an existing card's reaction value changes.
Tradeoffs: the picker is global for the current user rather than attached to each card, which matches the requested page-corner control but keeps card markup simpler. The browser accessibility tree may still expose hidden picker buttons when collapsed because they remain in the DOM; visually they are hidden and non-interactive.
Follow-up: if mobile width becomes tight, reduce emoji button sizes or make the tray horizontally scrollable.
</details>


### Reactions now auto-clear after 3 seconds
**What:** Participant reactions are now ephemeral: the server clears each user's reaction 3 seconds after it is set and broadcasts the removal to the room.
**Why:** The user clarified that reactions should behave like short-lived visual bursts, so client-side persistence was incorrect and cleanup needed to be authoritative and synchronized for all participants.
**Impact:** Reactions disappear automatically for everyone after 3 seconds, reaction state is no longer re-applied from localStorage on reload, and a regression test now verifies timed auto-clear behavior.

<details>
Context: After moving reactions to a floating messenger-style picker, the user requested that the emoji disappear after 3 seconds.
Options considered: clear on each client with a local timer, or clear on the server and broadcast the update.
Decision: implement a server-side 3-second timer per room/socket reaction in `index.js`, cancel/reset the timer when reactions change or the socket leaves, and remove client-side reaction persistence from `index.html`.
Tradeoffs: tests take about 3 extra seconds because the expiry is validated with the real timeout. The TTL is currently a hardcoded constant at 3000 ms.
Follow-up: if reaction timing needs to vary by environment later, expose the TTL through config.
</details>

### APP-12 adds file-backed estimate history
**What:** APP-12 stores revealed participant estimates in a JSON history file, exposes them through /api/estimation-history with field filters, and provides a dedicated /history/ page linked from the top-right room toolbar.
**Why:** The project has no database or existing persistence layer, so the smallest durable solution that matches the issue scope was a server-side JSON file plus a read-only HTML page for filtering and viewing history.
**Impact:** Revealed votes are now preserved across app restarts, users can open a separate history page from the main room UI, and targeted server tests cover both history recording and API filtering.

<details>
Context: Linear APP-12 required storing estimate history with task id, participant name, estimate, and estimate date, plus a separate page with filters over those fields. The existing app stored room state only in memory and had no backend persistence besides process state.
Options considered: keep history only in memory, add a database dependency, or persist a simple JSON file on the server.
Decision: implement a dedicated estimation-history-store module backed by a JSON file path configurable via ESTIMATION_HISTORY_FILE, record history entries when a room transitions from hidden votes to revealed, add /api/estimation-history for filtered reads, and serve a standalone history.html page from /history/.
Tradeoffs: history entries are recorded on reveal rather than on every interim vote change, which avoids noisy partial history but assumes reveal is the finalization moment. The stored schema intentionally matches the requested fields and does not add broader analytics or auth layers.
Follow-up: if history volume grows or multi-process writes become a concern, replace the JSON file with a database or append-only log format.
</details>
